### 2. DAG 스케줄 및 실행 제어

- DAG ID는 `ftp_batch_hourly`다.
- 스케줄은 `0 * * * *`로 매시간 정각 실행이다.
- Task는 `PythonOperator` 기반 단일 `COMBINED` 실행이다.
- `catchup=False`로 과거 실행 적재를 막았다.
- `max_active_runs=1`로 이전 실행이 끝나기 전에는 다음 실행이 겹치지 않게 고정했다.
- retry 설정은 `retries=1`, `retry_delay=5분`이다.

### 3. 배치 실행 흐름 연결

- DAG는 `run_combined(input_date="{{ ds }}")` 형태로 실행일자를 넘긴다.
- `BatchRunner`는 전달받은 `input_date`를 anchor로 사용하고, 내부적으로 `전날 + anchor` 두 날짜를 처리한다.
- `COMBINED` 흐름은 다음 순서로 동작한다.
  - `/RUBI/<date>` 와 `/RUIP/<date>` 스캔
  - 텍스트 기준 최근접 이미지 1:1 매칭
  - PNG 준비
  - 마지막 업로드 큐 일괄 업로드
  - 성공 시 DB 반영 후 FTP 원본 삭제

### 4. 검증 결과

- DAG, Airflow 래퍼, 배치 러너 포함 전체 Python 파일 `py_compile` 통과를 확인했다.
- 로컬 FTP 기반 수동 E2E 테스트를 수행했다.
- `COMBINED` 실행에서 다음 동작을 실제로 확인했다.
  - 전날 이미지 + 당일 텍스트 동시 조회
  - 텍스트 기준 최근접 이미지 매칭
  - `rbi/ruip/.../*.png` 업로드
  - 매칭 성공한 텍스트/이미지 FTP 원본 삭제
  - 매칭 없는 텍스트는 텍스트만 DB 저장
  - 로컬 raw `txt`/`tif` 삭제, 로컬 `png` 캐시 유지

## 핵심 변경

- Airflow 공식 진입점을 `COMBINED` 단일 태스크로 고정했다.
- 운영 실행 제어는 `hourly + max_active_runs=1` 기준으로 정리했다.
- Airflow에서 넘기는 실행일자와 실제 배치 내부의 `전날 + anchor` 처리 규칙을 연결했다.
- 검증은 문법 확인이 아니라 로컬 FTP 기반 end-to-end까지 포함해 확인했다.

## 실행 제어 포인트

- DAG는 한 번에 하나만 돈다.
- 배치 내부는 단일 `input_date`로 두 날짜를 처리한다.
- Airflow는 orchestration만 담당하고, 실제 파일 처리와 매칭 로직은 `BatchRunner`가 담당한다.
